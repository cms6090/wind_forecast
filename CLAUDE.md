# CLAUDE.md — 제3회 풍력발전량 예측 AI 경진대회 (BARAM 2026)

이 문서는 이 저장소에서 작업하는 모든 AI 어시스턴트(Claude Code, Codex 등)가 반드시 따라야 하는 총괄 규칙이다.

## 1. 프로젝트 개요

- **목표**: 강원도 태백 가덕산 풍력발전단지의 KPX 그룹 3개(`kpx_group_1/2/3`)에 대해 2025년 전체 기간의 **시간별 발전량(kWh)** 예측
- **평가지표**: `Score = 0.5 × (1 - NMAE) + 0.5 × FICR`
  - 실제 발전량이 설비용량의 10% 이상인 시간대만 평가
  - FICR: 오차율 ≤6% → 4원/kWh, ≤8% → 3원/kWh, 초과 → 0원 (계단 함수)
- **설비용량(1시간 환산)**: group_1 = 21,600 kWh, group_2 = 21,600 kWh, group_3 = 21,000 kWh
- **제출 마감**: 2026-08-14(금) 10:00, 일일 최대 5회 제출
- **2차 평가 요건**: train/inference 코드 분리, Private Score 재현 가능, 외부 API 추론 금지, pretrained 모델은 2026-07-05 이전 공개 가중치만 허용

## 3. 진행 로드맵 (반드시 이 순서)

1. 문제와 평가 지표 확인 → `src/metric.py`, `src/submission.py` 셋업
2. 학습·검증 데이터 분리 — **시계열이므로 과거로 학습, 미래로 검증** (전처리 전에 분리 설계)
3. EDA — 결측/이상치/분포/타깃 관계/시간 순서/그룹 구조
4. 기본 전처리 — `01_preprocessing.ipynb`, train+test 동일 로직 → `data/processed/` parquet 캐시
5. 베이스라인 — 단순 모델(평균, 선형회귀, LightGBM 기본값)로 비교 기준 확립
6. 피처 엔지니어링 및 선택 — 도메인 근거 기반 (`wind-domain-features` 스킬 참조)
7. 모델 선택 — 동일한 검증 방식으로 비교
8. 하이퍼파라미터 튜닝 — 유망 모델만, 동일 CV 구조 유지
9. 최종 검증 및 앙상블 — seed 안정성, 오류 분석
10. 전체 학습 데이터로 재학습 후 예측

## 4. 절대 규칙 (위반 금지)

### 데이터 누수 (가장 중요)
- **예측기준시점 원칙**: 각 예측값은 그 시점 이전에 실제로 활용 가능했던 정보만 사용. 예보 데이터는 `data_available_kst_dtm` 기준으로 판단
- 평가 기간의 실제 발전량·SCADA·재분석자료 사용 금지
- 파생 변수/통계값/보간값에도 미래 정보 포함 금지
- 모든 피처 생성 시 `leakage-guard` 스킬로 자가 점검

### metric.py
- `src/metric.py`는 대회 공식 산식 그대로. **절대 수정하지 않는다**

### Git
- `git add / commit / push`는 민석님이 직접 실행한다. AI는 커밋 타이밍과 메시지 문구를 **제안만** 한다
- 저장소 명칭·파일명은 ASCII만 사용

### 실행 방식
- 셀 실행은 민석님이 직접 한다. AI는 코드를 작성/수정하고, 실행 결과(shape, head() 등)를 받은 뒤 다음 단계를 안내한다
- 무거운 연산을 자율 실행하지 않는다

## 5. 작업 방식

- **노트북 우선**: 주 작업 공간은 `.ipynb`. 필수 `.py`는 `src/metric.py`, `src/submission.py`뿐
- **가상환경**: 모든 패키지는 `venv` 안에서만 설치. 새 패키지 설치 시 `requirements.txt` 갱신 제안
- **근거 기반**: 모든 전처리·피처·모델 결정은 논문, 풍력 도메인 지식, 데이터 근거를 명시한다. "그냥 해보자"는 금지. 근거 예: 발전량 ∝ 풍속³, 파워커브, 윈드시어 멱법칙 등
- **중간 결과 확인**: 각 단계마다 shape 변화, 샘플 출력, 분포를 확인하고 진행
- **설명 언어**: 모든 설명은 평이한 한국어. 기술 용어는 괄호 안에 정의나 비유로 인라인 설명 (세부 규칙은 아래 5.1)
- **보고서**: 각 단계 완료 시 `reports/`에 Why / How / Result / So-what 구조로 문서화. How는 처리 단계 하나당 하위 항목 하나

### 5.1 설명 스타일 — 쉽고 친절하게

민석님은 모든 과정을 직접 이해하면서 진행한다. AI는 전문가끼리 대화하듯 쓰지 말고, **처음 배우는 사람에게 과외하듯** 설명한다.

- **결론 먼저, 이유는 그다음**: "이번엔 결측치를 채울 거예요. 왜냐하면 ~" 순서로. 무엇을 → 왜 → 어떻게
- **전문 용어는 그냥 쓰지 않는다**: 처음 등장하는 용어는 반드시 한 줄 정의 + 일상 비유를 붙인다
  - 예: "리샘플링(10분마다 찍힌 데이터를 1시간 단위로 합치는 것 — 10분 단위 가계부를 월별 가계부로 합산하는 느낌)"
  - 예: "early stopping(검증 점수가 더 이상 좋아지지 않으면 학습을 멈추는 것 — 성적이 안 오르면 야자 그만하는 것)"
- **코드 셀마다 "이 셀이 하는 일"을 사람 말로 먼저**: 코드보다 마크다운 설명이 먼저 온다. 코드 안에도 줄 단위 주석
- **수식·산식은 말로 풀어서 한 번 더**: 예: "NMAE = 오차를 설비용량으로 나눈 것. 즉 '전체 용량 대비 몇 % 틀렸나'라는 뜻"
- **선택지가 있으면 비교표 + 추천 + 이유**: "A안/B안이 있고, 저는 A안을 추천해요. 이유는 ~" 형태. 일방적으로 결정하지 않는다
- **한 번에 하나씩**: 긴 설명은 단계로 쪼개고, 단계마다 "여기까지 이해되셨으면 다음으로 갈게요" 흐름 유지
- **어려운 개념은 스스로 확인 질문 제공**: 설명 끝에 "이해 체크: ~를 한 문장으로 말할 수 있으면 OK" 한 줄
- 단, 쉬운 설명이 **정확성을 희생하면 안 된다**: 비유가 개념과 어긋나는 부분이 있으면 "정확히는 ~"로 보정한다

## 6. 세션 연속성 (학교 ↔ 집)

- 매 세션 종료 시 `HANDOFF.md`를 갱신한다 (`session-handoff` 스킬 참조)
- 새 세션 시작 시 AI는 가장 먼저 `HANDOFF.md`를 읽고 현재 상태를 요약한 뒤 시작한다
- 커밋 제안 시 HANDOFF.md 갱신 여부를 함께 확인한다

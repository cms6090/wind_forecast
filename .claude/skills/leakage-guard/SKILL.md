# Leakage Guard — 예측기준시점 원칙

이 대회의 실격 사유 1순위는 데이터 누수다. 모든 데이터 조작은 아래 원칙을 통과해야 한다.

## 핵심 원칙

각 예측값에는 **예측기준시점 이전에 생성·공개·확정되어 실제로 활용 가능했던 정보만** 사용할 수 있다.

- 판단 기준은 데이터가 **가리키는 시각**이 아니라 데이터가 **활용 가능해진 시각**이다
- 예보 데이터: `data_available_kst_dtm` 기준. 매일 09:00 KST 초기화 예보가 당일 13:00부터 사용 가능하며, 다음날 01:00 ~ 그다음날 00:00의 24시간을 커버한다
- 실측 데이터(SCADA 등): 예측기준시점 이전에 이미 관측·확정된 과거 자료만 사용 가능

## 이 대회의 운영 구조 (실무 시나리오)

D-1일 13:00에 예보가 나오고, 그 예보로 D일 01:00~D+1일 00:00을 예측한다.
즉 **예측 대상 시각으로부터 최소 12~36시간 전의 정보만 존재한다**고 생각하면 안전하다.

## 피처별 판정 가이드

| 피처 유형 | 판정 | 이유 |
|---|---|---|
| test 기간 GFS/LDAPS 예보값 | ✅ 허용 | data_available_kst_dtm이 예측기준시점 이전 |
| SCADA 실측 풍속/발전량 (train 기간) | ⚠️ 조건부 | 학습 시 타깃 보조·분석용은 가능하나, **test 기간에는 존재하지 않으므로 추론 피처로 쓸 수 없다**. train에만 있는 피처는 만들지 않는다 |
| 발전량 lag 피처 (예: 1시간 전 발전량) | ❌ 금지 | test 기간 실제 발전량은 미제공. 예측기준시점(D-1 13:00) 기준으로도 D일 새벽 발전량은 미래 |
| 발전량 lag ≥ 36시간 + 재귀 예측 | ⚠️ 위험 | 이론상 가능하나 test에서 자기 예측을 입력해야 해 오차 누적. 기본적으로 배제 |
| 전체 기간 평균/표준편차로 스케일링 | ❌ 금지 | validation/test 정보가 train 통계에 섞임. **train 통계로만 fit 후 transform** |
| 시간 피처 (hour, month, sin/cos) | ✅ 허용 | 달력 정보는 항상 알 수 있음 |
| 예보값의 시간별 diff/rolling (같은 발표분 내) | ✅ 허용 | 같은 data_available_kst_dtm 내 24시간은 동시에 공개됨 |
| 미래 시각 예보값 피처 (예: 3시간 뒤 예보 풍속) | ✅ 허용 | 같은 발표분이면 이미 공개된 정보. 단, 발표분 경계를 넘지 않는지 확인 |
| target encoding | ⚠️ 위험 | 반드시 CV fold 내부에서만 계산. 전체 데이터로 계산 시 누수 |
| 결측 보간 | ⚠️ 조건부 | 미래 값을 쓰는 보간(interpolate 양방향, bfill)은 test 상황에서 재현 불가능하면 금지. ffill 또는 같은 발표분 내 보간만 |

## 검증 분리 규칙

- 시계열이므로 **시간 순서 분리만** 허용: 과거 → 학습, 미래 → 검증
- 랜덤 셔플 K-Fold 금지
- 전처리 파이프라인(스케일러, 인코더, 결측 대치 통계)은 **분리 후 train 파트에만 fit**

## 자가 점검 체크리스트 (피처 하나 만들 때마다)

1. 이 값은 예측기준시점(대상일 전날 13:00)에 존재했는가?
2. test 기간에 **같은 방식으로** 계산 가능한가? (train에만 있는 원천이면 탈락)
3. 통계량(평균, 분위수 등)을 썼다면 train 파트로만 계산했는가?
4. 보간·집계에 미래 행이 섞이지 않았는가?
5. 2차 평가에서 이 피처의 활용 가능 시점을 말로 소명할 수 있는가?

하나라도 "아니오"면 그 피처는 폐기하거나 재설계한다. 애매하면 민석님에게 판정 근거와 함께 보고하고 결정을 받는다.

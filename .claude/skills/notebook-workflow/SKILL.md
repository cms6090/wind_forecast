# Notebook Workflow — 노트북 작성 규약

## 1. 기본 원칙

- 주 작업 공간은 `.ipynb`. `.py`로 전환하지 않는다 (예외: `src/metric.py`, `src/submission.py`)
- **셀 실행은 민석님이 직접 한다.** AI는 셀 코드를 작성하고, 각 셀 아래에 "실행 후 확인할 것"(예상 shape, 확인 포인트)을 마크다운으로 명시한 뒤, 실행 결과를 받고 다음 단계로 진행한다
- 한 셀은 한 가지 일만: 로드 / 변환 / 확인 출력을 섞지 않는다
- 모든 설명 마크다운은 평이한 한국어. 기술 용어는 괄호로 정의 (예: "리샘플링(10분 데이터를 1시간 단위로 합치는 것)")

## 2. 노트북 공통 구조

모든 노트북은 아래 순서를 따른다:

1. **제목 + 목적** 마크다운 (이 노트북의 Why, 입력/출력 명시)
2. **셋업 셀**: import, 경로 상수, `pd.set_option`, 시드 고정
3. **본문**: 단계별 (마크다운 설명 → 코드 셀 → 확인 셀) 반복
4. **저장 셀**: 산출물을 `data/processed/`에 parquet 저장
5. **요약 마크다운**: 이 노트북이 결정한 것, 다음 노트북으로 넘기는 것

## 3. A방식 — parquet 캐시 파이프라인

- `01_preprocessing.ipynb`에서 **train과 test를 동일한 셀 로직**으로 처리한다 (함수로 정의 후 두 번 호출)
- 결과를 `data/processed/*.parquet`으로 저장 → 이후 노트북은 캐시만 로드
- 파일명 규칙: `train_base.parquet`, `test_base.parquet`, `train_features_v1.parquet` 등 버전 명시
- `train.ipynb` / `inference.ipynb`는 각자 캐시를 로드해 독립 실행 가능해야 한다 (2차 평가 분리 요건)
- 캐시 로직을 바꾸면 하위 노트북 재실행 필요 — 변경 시 반드시 어떤 캐시가 무효화되는지 알려준다

## 4. 확인 출력 규약 (매 변환 후)

```python
print(df.shape)            # 행/열 변화
display(df.head(3))        # 샘플
print(df.isna().sum().sum())  # 결측 총량
```

시간 인덱스를 만졌으면 추가로: 기간 min/max, 중복 여부, 시간 간격 검사.

## 5. reports/ 보고서 규칙

각 노트북 완료 시 같은 번호의 보고서를 작성한다 (예: `reports/01_preprocessing.md`).

구조 (고정):
- **Why**: 왜 이 단계가 필요한가
- **How**: 처리 단계 하나당 하위 항목 하나. 각 항목에 근거(도메인/논문/EDA 결과) 명시
- **Result**: 수치·표·그림으로 결과 요약
- **So-what**: 이 결과가 다음 단계 결정에 주는 함의

## 6. 환경 규칙

- 모든 실행은 `venv` 활성화 상태 전제. 패키지 설치 제안 시 `pip install` 명령과 함께 `requirements.txt` 갱신을 안내
- 노트북 상단에서 `sys.executable` 확인 셀 제공 (venv 커널인지 검증)
- 시드 고정: `SEED = 42`를 셋업 셀에 상수로, numpy/모델에 일괄 적용
- git 관련: 노트북 출력이 큰 diff를 만들므로, 커밋 제안 시 "출력 지우고 커밋할지" 여부를 함께 제안한다 (실행은 민석님 담당)
